# Chapter 03 - In-Memory Product API

## Why Now

The first Axum server works. Now make it slightly more maintainable before
adding a database.

## Learn First

- [Serde JSON](https://serde.rs/json.html)
- [Axum extractors](https://docs.rs/axum/latest/axum/extract/index.html)

## New Idea

Separate HTTP handlers from product behavior.
Add route constants and a central config struct before the API grows.
Add a small ID/time contract before persistence makes identity and timestamps
harder to reason about. Error handling and tracing are first-class chapters
immediately after this one.

## Alternatives

Choice:

- Keep product decisions outside handlers.

Pros:

- Product behavior is easier to test.
- Handlers stay readable.
- Later database changes do not rewrite every route.

Cons:

- More files than a small demo.
- Requires naming discipline.

Why this is right now:

- The API is still small, so this is the best time to build the habit before
  persistence and GraphQL add more complexity.

## Why We Choose This

We choose thin handlers plus domain functions because the same product behavior
will later be reached from HTTP, GraphQL, tests, workers, and compatibility code.

## What Engineers Should Notice

Handlers are transport adapters. They should parse requests, call the right
domain flow, and return responses.

```text
route constant -> handler -> domain function -> response DTO
```

Central config and route constants prevent hidden string/config drift before the
project grows.

Public API responses, IDs, and timestamps are contracts. Keep them stable before
database, GraphQL, workers, and cache behavior make them harder to change.

## Chapter 03 Tasks

### Task 03.1 - Create The Public Folder Shape

Input:

- Axum API from chapter 02

Output:

```text
apps/api/
packages/catalog/
```

Done when:

- API still starts after the move
- the folder shape does not mirror the private Majaz repo

### Task 03.2 - Add Constants And Config

Input:

- working API routes

Output:

- route constants for `/health` and product routes
- env key constants
- central `Config` loaded once at startup
- `AppState` receives `Config`

Done when:

- handlers do not hardcode route strings or read env directly

### Task 03.3 - Add DTOs And Handler Boundaries

Input:

- product domain behavior
- route/config constants

Output:

- request DTO for product create
- response DTO for product list/create
- handlers that convert DTOs and call domain functions
- unit tests for domain behavior
- HTTP tests or documented curl checks

Done when:

- API input/output matches
  [Chapter 02 API contract](../02-axum-basics/api-contract.md)
- you can explain handler vs domain logic

### Task 03.4 - Define IDs And Time

Input:

- product create/list behavior

Output:

- ID/time rules based on [id-time-contract.md](id-time-contract.md)
- product IDs generated by Rust as UUIDv7 values
- `created_at` and `updated_at` produced by an application clock boundary
- tests that can use fixed IDs or a fixed clock

Done when:

- no database-generated IDs are required by the product flow
- tests are deterministic

## Do Not Add Yet

- database
- Makefile
- mprocs
- query code generation

## Done When

- API still works.
- Product behavior tests pass.
- You can explain handler vs domain logic.
- You can explain why route strings and config values are centralized.
